# 流水线冲突、相关与冒险

> 来源依据：`知识点汇总.txt` 第七章第 2 点；第 7 章 PPT“指令流水线的冲突、相关、冒险”“数据相关处理机制”“数据相关检测逻辑”；教材第 7 章。

## 一句话理解

流水线冒险就是“下一条指令本来想继续前进，但如果现在前进，就可能拿不到资源、拿错数据或走错分支方向”。

## 基本概念

### 1. 相关和冒险不是一回事

先区分两个词：

| 概念 | 含义 |
| --- | --- |
| 相关 | 程序语义上存在依赖关系 |
| 冒险 | 这种依赖在流水线硬件中会导致错误或必须停顿 |

例如：

```mips
add $s1, $s2, $s3
sub $s4, $s1, $s5
```

第二条 `sub` 使用第一条 `add` 写出的 `$s1`，这叫数据相关。若流水线中 `sub` 需要 `$s1` 时，`add` 的新值还没写回，就会形成数据冒险。

### 2. 三类流水线冒险

| 类型 | 根本原因 | 直觉 | 例子 |
| --- | --- | --- | --- |
| 结构冒险 | 硬件资源不够 | 两条指令抢同一台机器 | 取指和数据访存共用同一存储器 |
| 数据冒险 | 数据还没准备好 | 后一道工序要用前一道工序的结果 | `add` 后紧跟使用结果 |
| 控制冒险 | 下一条指令地址不确定 | 不知道下一张任务单该拿哪张 | `beq`、`j`、异常中断 |

做题时先判断是哪一类冒险，再考虑解决方法。不要看到停顿就默认是数据冒险。

### 3. 数据相关的 RAW、WAR、WAW

| 类型 | 含义 | 例子 |
| --- | --- | --- |
| RAW | 先写后读：后一条要读前一条写的值 | `add $1,...` 后 `sub ..., $1,...` |
| WAR | 先读后写：后一条写了前一条还没读的寄存器 | 乱序执行中可能出现 |
| WAW | 先写后写：两条指令写同一寄存器，写回顺序不能反 | 乱序执行中可能出现 |

经典按序五段 MIPS 中最常见、最需要掌握的是 RAW。因为指令按序进入流水线，寄存器读通常在 ID，写回在 WB，WAR 和 WAW 在这种简单流水线中一般不会成为主要问题。

## 工作原理或推导过程

### 1. 判断数据冒险的核心问题

不要机械背“隔几条指令”。更可靠的方法是问两句话：

```text
前一条指令什么时候产生结果？
后一条指令什么时候需要这个结果？
```

对五段流水线：

| 指令结果来源 | 结果大致什么时候可用 |
| --- | --- |
| ALU 运算结果 | EX 阶段结束后 |
| `lw` 读出的数据 | MEM 阶段结束后 |
| 寄存器正式写回 | WB 阶段 |

后一条 ALU 类指令通常在 EX 阶段真正需要操作数。它在 ID 阶段读寄存器只是先取一次值，若这个值旧了，后面可以通过转发修正。

### 2. ALU 后接 ALU：可用转发解决

```mips
add $s1, $s2, $s3
sub $s4, $s1, $s5
```

`add` 的结果在 EX 末尾已经由 ALU 算出来。下一条 `sub` 在自己的 EX 阶段需要 `$s1`。这时可以不等 `add` 到 WB 写回，而是把 `add` 后面流水寄存器中的 ALU 结果直接送到 `sub` 的 ALU 输入。

这叫数据转发或旁路。

直觉是：成品已经从机器里出来了，只是还没放回货架；下一道工序可以直接从机器出口拿，不必等它入库。

### 3. load-use：通常仍要插入气泡

```mips
lw  $s1, 0($s0)
add $s2, $s1, $s3
```

`lw` 的数据要到 MEM 阶段结束后才从数据存储器出来，而紧跟着的 `add` 在 EX 阶段就要用 `$s1`。如果不暂停，`add` 需要数据时，数据还没从存储器出来。

即使有转发，也通常需要 1 个气泡：

```mips
lw  $s1, 0($s0)
nop
add $s2, $s1, $s3
```

气泡不是为了“修正数据”，而是为了“拖延时间”，让数据来得及到达。

### 4. 结构冒险怎么解决

结构冒险来自资源争用。例如只有一个统一存储器时：

- IF 阶段要取指令。
- MEM 阶段要读写数据。

如果两条指令在同一周期分别需要这两个动作，就会抢同一个存储器。

解决方法：

- 增加资源，例如指令存储器和数据存储器分离，或使用 I-Cache 和 D-Cache。
- 插入停顿，让其中一条晚一点使用资源。
- 改变数据通路设计，提高并发性。

### 5. 控制冒险怎么解决

分支指令的问题是：在分支是否跳转确定之前，IF 阶段可能已经取了后续指令。如果后来发现方向错了，就要清除误取的指令。

常见方法：

| 方法 | 思路 |
| --- | --- |
| 停顿等待 | 直到分支结果确定再取后续指令 |
| 提前判断分支 | 在 ID 阶段尽早比较和计算目标地址 |
| 冲刷流水线 | 分支确定后清除误取指令 |
| 分支延迟槽 | 规定分支后一条指令无论是否跳转都执行 |
| 分支预测 | 先猜方向，猜错再冲刷 |

分支越早确定，损失越小。若在 EX 阶段才确定，前面可能已经取入多条错误路径指令。

### 6. 数据冒险检测的直觉

硬件检测数据冒险时，本质是在比较：

```text
当前 ID 阶段指令要读的寄存器
是否等于 EX/MEM 等后续阶段将要写回的目标寄存器
```

简化理解：

```text
如果 ID.rs 或 ID.rt 正在被使用
并且前面某条尚未写回的指令会写同一个非 0 寄存器
就可能有数据冒险
```

对带转发的流水线，最典型需要停顿的是 load-use：

```text
LoadUse = EX.MemRead
          且 EX.WriteReg != 0
          且 ID 阶段确实要用 EX.WriteReg
```

其中 `WriteReg != 0` 是因为 MIPS 的 `$0` 恒为 0，写 `$0` 没有实际意义。

## 典型例子

### 例 1：判断是否有 RAW 相关

```mips
add $1, $2, $3
or  $4, $1, $5
```

第一条写 `$1`，第二条读 `$1`，所以有 RAW 相关。若无转发，需要等写回；若有转发，通常可直接转发解决。

### 例 2：load-use 冒险

```mips
lw  $2, 4($5)
and $1, $2, $3
```

`lw` 写 `$2`，`and` 读 `$2`。`lw` 的数据到 MEM 后才可用，而 `and` 太早进入 EX，需要插入 1 个气泡。

### 例 3：不是所有相邻指令都有冒险

```mips
add $1, $2, $3
sub $4, $5, $6
```

第二条没有读 `$1`，也没有抢特殊资源或改变控制流，所以和第一条之间没有数据冒险。

## 常见考法

- 判断指令序列中是否存在结构、数据、控制冒险。
- 区分 RAW、WAR、WAW。
- 判断某个数据相关能否用转发解决。
- 判断 load-use 是否需要插入气泡。
- 给指令序列画带停顿或气泡的时空图。
- 根据分支执行阶段判断需要冲刷几条指令。
- 写出或理解简化的数据冒险检测条件。

## 易错点

- 数据相关是语义关系，数据冒险是流水线中会导致错误的实际风险。
- 有相关不一定必须停顿；有转发时很多 RAW 可以不阻塞。
- 转发不是提前写回寄存器，而是把中间结果直接送到需要它的部件。
- load-use 即使有转发也常要 1 个气泡，因为 load 数据产生太晚。
- 结构冒险是资源不够，不是数据值不对。
- 控制冒险来自下一条指令地址不确定，不只 `beq`，跳转和异常中断也可能导致控制流改变。
- `$0` 寄存器特殊，判断写回相关时通常排除目标寄存器为 0 的情况。

## 小结

- 流水线冒险分为结构冒险、数据冒险、控制冒险。
- 五段 MIPS 中最常见的数据冒险是 RAW。
- 判断数据冒险的核心是“结果什么时候产生，使用者什么时候需要”。
- 转发能解决很多 ALU 相关，load-use 通常仍需气泡。
- 控制冒险要靠提前判断、冲刷、延迟槽或分支预测降低损失。
