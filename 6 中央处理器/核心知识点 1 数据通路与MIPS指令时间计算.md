# 数据通路与 MIPS 指令时间计算

> 来源依据：`知识点汇总.txt` 第六章；第 6 章 PPT“数据通路”“单总线结构 CPU”“单周期 MIPS”“多周期 MIPS”；教材第 6 章。

## 一句话理解

数据通路解决“数据从哪里来、经过哪里、送到哪里去”，时间计算解决“这些动作分几个时钟节拍完成，每个节拍有多长”。

## 基本概念

初学时不要先背控制信号，先把 CPU 想成一个“会按步骤搬东西和算东西的车间”：

- 寄存器像临时货架，保存地址、操作数、运算结果。
- ALU 像加工机器，负责加、减、比较等运算。
- 存储器像仓库，按地址读写数据。
- 总线或连线像通道，决定数据能不能从一个部件走到另一个部件。
- 控制器像调度员，决定某个时钟周期哪些门打开、哪个部件工作、结果锁存到哪里。

所以，**数据通路**不是一条固定的线，而是 CPU 内部所有可能传送路径的总称。执行不同指令时，控制器会打开不同的路径。例如 `add` 主要走“寄存器 -> ALU -> 寄存器”，`lw` 还要多走“ALU 算地址 -> 数据存储器 -> 寄存器”。

### 数据流和控制流

一条指令真正执行时有两条线同时存在：

| 名称 | 问的问题 | 例子 |
| --- | --- | --- |
| 数据流 | 数据实际怎么移动 | `R[rs]` 送入 ALU，ALU 结果写回 `R[rd]` |
| 控制流 | 谁让数据这样移动 | `RegWrite=1`、`ALUSrc=0`、`MemtoReg=0` |

做题时常见的错误是只看数据流，不看控制信号。例如结果能不能写回寄存器，不是看 ALU 有没有算出来，而是看写回阶段 `RegWrite` 是否有效。

### 单总线、单周期、多周期的直觉区别

| 实现方式 | 直觉类比 | 关键特点 |
| --- | --- | --- |
| 单总线结构 | 全车间共用一条通道 | 结构简单，但同一时刻通常只能有一个部件把数据送上总线，并发性差 |
| 单周期处理器 | 每条指令必须一次性走完全流程 | `CPI=1`，但一个周期必须长到足够容纳最慢指令 |
| 多周期处理器 | 一条指令拆成多个较短步骤 | 不同指令 CPI 不同，时钟周期可以比单周期短 |

单周期不是“每条指令只干一步”，而是“一个很长的时钟周期里把所有必要步骤都做完”。多周期不是“更慢”，而是把一条大路切成多个小步骤，简单指令少走几步，复杂指令多走几步。

## 工作原理或推导过程

### 1. 先看一条指令到底需要哪些动作

以 MIPS 常见指令为例：

| 指令 | 指令含义 | 必经动作 |
| --- | --- | --- |
| `add rd, rs, rt` | `R[rd] = R[rs] + R[rt]` | 取指、译码读寄存器、ALU 运算、写回 |
| `addi rt, rs, imm` | `R[rt] = R[rs] + SignExt(imm)` | 取指、译码读寄存器、立即数扩展、ALU 运算、写回 |
| `lw rt, imm(rs)` | `R[rt] = M[R[rs] + SignExt(imm)]` | 取指、译码读寄存器、ALU 算地址、读数据存储器、写回 |
| `sw rt, imm(rs)` | `M[R[rs] + SignExt(imm)] = R[rt]` | 取指、译码读寄存器、ALU 算地址、写数据存储器 |
| `beq rs, rt, imm` | 若相等则跳转 | 取指、译码读寄存器、比较、计算分支目标地址、必要时改 PC |

这里最重要的理解是：**时间长短不是由指令名字决定的，而是由它需要经过多少硬件动作决定的**。`lw` 往往慢，是因为它既要算地址，又要访问数据存储器，还要写回寄存器。

### 2. 单总线结构为什么通常周期数多

单总线结构中，很多部件都挂在同一条内部总线上。为了避免两个部件同时向总线输出造成冲突，一个节拍里通常只允许一个主要数据源输出。

例如取指在课件中的节拍可以理解为：

| 节拍 | 数据动作 | 为什么不能随便合并 |
| --- | --- | --- |
| T1 | `PC -> AR`，同时把 `PC` 送入临时寄存器 `X` | 要用 PC 作为取指地址，也要准备计算 `PC+4` |
| T2 | `X + 4 -> Z` | ALU 计算下一条指令地址 |
| T3 | `Z -> PC`，同时 `M[AR] -> DR` | 更新 PC，并把指令从存储器读入数据寄存器 |
| T4 | `DR -> IR` | 把取到的指令装入指令寄存器，供译码使用 |

对单总线 MIPS，课件中给出的执行部分节拍大致是：

| 指令 | 取指节拍 | 执行节拍 | 合计节拍 |
| --- | ---: | ---: | ---: |
| `lw` | 4 | 5 | 9 |
| `sw` | 4 | 5 | 9 |
| `beq` | 4 | 5 | 9 |
| `add` | 4 | 3 | 7 |
| `addi` | 4 | 3 | 7 |

这里的“合计节拍”不是死记硬背出来的，而是数“必须串行走几次总线、几次 ALU、几次访存”。如果题目给了不同的数据通路图，要按图重新数。

### 3. 单周期处理器如何算时间

单周期处理器规定：

```text
每条指令用 1 个时钟周期完成
CPI = 1
```

但它的问题在于：时钟周期必须按最慢指令设置。对于典型 MIPS 单周期数据通路，`lw` 通常是关键路径：

```text
PC
-> 指令存储器取指
-> 寄存器堆读 rs
-> ALU 计算有效地址
-> 数据存储器读数据
-> 选择写回数据
-> 寄存器堆写回
```

因此单周期的最小时钟周期通常按下面思想估计：

```text
Tclk >= Tclk_to_q + T关键路径组合逻辑 + Tsetup
```

如果题目把各部件延迟拆开给出，则单周期的时钟周期要取“最慢指令完整路径”的总延迟，而不是取某一个阶段的最大值。

执行时间公式为：

```text
执行时间 = 指令条数 * CPI * 时钟周期
        = 指令条数 * 1 * 单周期时钟周期
```

单周期 `CPI=1` 看起来很漂亮，但它牺牲了时钟周期长度。很多简单指令本来不需要访问数据存储器，也被迫等一个很长的周期结束。

### 4. 多周期处理器如何算时间

多周期处理器把指令拆成多个阶段。每个阶段只做一部分工作，例如：

```text
取指 -> 译码/读寄存器 -> 执行或地址计算 -> 访存 -> 写回
```

因此，多周期有两个关键变化：

1. 时钟周期按“最长阶段”设置，而不是按“最慢整条指令”设置。
2. 不同指令走的阶段数不同，所以 CPI 不再固定为 1。

典型 MIPS 多周期常见周期数可按下面理解：

| 指令 | 常见周期数 | 原因 |
| --- | ---: | --- |
| `lw` | 5 | 取指、译码、算地址、读存储器、写回 |
| `sw` | 4 | 取指、译码、算地址、写存储器，不需要写回寄存器 |
| R 型如 `add` | 4 | 取指、译码、ALU 运算、写回 |
| `addi` | 4 | 取指、译码、ALU 加立即数、写回 |
| `beq` | 3 | 取指、译码、比较并更新 PC |

若题目给的是课件单总线节拍表，就按课件节拍数算；若题目给的是标准多周期 MIPS，就按题目说明的阶段数算。不要把“单总线节拍数”和“标准多周期五阶段数”混在一起。

平均 CPI 的计算来自加权平均：

```text
平均 CPI = Σ(某类指令比例 * 该类指令周期数)
执行时间 = 指令条数 * 平均 CPI * 多周期时钟周期
```

为什么是加权平均？因为一个程序不是只由一种指令组成。`lw` 多一点，平均 CPI 就被拉高；R 型多一点，平均 CPI 可能更低。

### 5. 计算题的固定步骤

遇到第六章时间计算，建议按这个顺序做：

1. 判定处理器类型：单总线、单周期、多周期，还是题目给定的数据通路。
2. 求时钟周期：单周期看最慢整条指令；多周期看最长阶段；单总线常看一个节拍的最长微操作。
3. 求 CPI：单周期为 1；多周期要按各类指令周期数和比例加权。
4. 代入总公式：

```text
CPU 时间 = 指令条数 * 平均 CPI * 时钟周期
CPU 时间 = 指令条数 * 平均 CPI / 主频
```

## 典型例子

### 例 1：多周期平均 CPI

某程序中各类指令比例为：

| 指令类型 | 比例 | 周期数 |
| --- | ---: | ---: |
| `lw` | 10% | 5 |
| `sw` | 10% | 4 |
| `beq` | 10% | 3 |
| R 型 | 50% | 4 |
| I 型运算 | 20% | 4 |

则：

```text
平均 CPI
= 0.10*5 + 0.10*4 + 0.10*3 + 0.50*4 + 0.20*4
= 0.5 + 0.4 + 0.3 + 2.0 + 0.8
= 3.7
```

如果程序共有 `1,000,000` 条指令，时钟周期为 `2ns`：

```text
执行时间 = 1,000,000 * 3.7 * 2ns
        = 7,400,000ns
        = 7.4ms
```

### 例 2：为什么单周期不一定更快

假设单周期中 `lw` 是最慢指令，完整路径延迟为 `500ps`，所以：

```text
单周期：CPI = 1，Tclk = 500ps
每条指令平均时间 = 1 * 500ps = 500ps
```

多周期把路径拆开后，最长阶段只需 `150ps`，但平均 CPI 为 `3.7`：

```text
多周期每条指令平均时间 = 3.7 * 150ps = 555ps
```

这个例子中多周期反而略慢。若多周期时钟周期能降到 `120ps`：

```text
3.7 * 120ps = 444ps
```

多周期就更快。结论是：**不能只看 CPI，也不能只看时钟周期，必须看二者乘积**。

### 例 3：单总线节拍数

若课件式单总线结构中取指固定 4 个节拍，`add` 执行部分 3 个节拍，则：

```text
add 总节拍数 = 4 + 3 = 7
```

若一个程序有 `1000` 条 `add`，每个节拍为 `10ns`：

```text
执行时间 = 1000 * 7 * 10ns = 70,000ns = 70us
```

如果换成 `lw`，总节拍数变为 `9`，不是因为 `lw` 的名字特殊，而是因为它多了“访存读数据”和“写回寄存器”等动作。

## 常见考法

- 给部件延迟，求单周期处理器最小时钟周期。
- 给各阶段延迟，求多周期处理器最小时钟周期。
- 给指令比例和各类指令 CPI，求平均 CPI 与执行时间。
- 根据单总线数据通路图，写出某条指令每个节拍的数据流和控制信号。
- 比较单周期、多周期、单总线结构的性能差异。
- 判断某个控制信号恒 0 或恒 1 会影响哪些指令。

## 易错点

- `CPI=1` 不等于性能一定最好；单周期时钟周期可能非常长。
- 多周期的时钟周期不是所有阶段延迟之和，而是最长阶段延迟加必要的寄存器开销。
- 平均 CPI 必须按指令比例加权，不能把各类 CPI 简单相加再除以种类数。
- 单总线题要注意“同一节拍谁能上总线”，不能把两个互斥输出信号放在同一节拍。
- `sw` 不需要写回寄存器，`lw` 需要写回寄存器。
- `beq` 的分支目标地址通常基于 `PC+4` 加偏移量，而不是旧 PC 直接加偏移量。
- 控制信号故障题要看“本来哪些指令需要该信号”，不要只看某一条指令。

## 小结

- 数据通路是 CPU 内部数据移动和加工的路径，控制器负责按时打开这些路径。
- 单总线结构简单，但共享通道会降低并发性，节拍数常较多。
- 单周期处理器 `CPI=1`，但时钟周期由最慢指令决定。
- 多周期处理器时钟周期较短，但不同指令 CPI 不同。
- 性能计算统一回到公式：`执行时间 = 指令条数 * 平均 CPI * 时钟周期`。
