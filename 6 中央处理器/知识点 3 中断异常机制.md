# 中断与异常机制

> 来源依据：第 6 章 PPT“中断机制实现”“异常与中断”“中断异常处理数据通路升级”“支持中断的微程序控制器”；教材第 6 章。

## 一句话理解

中断和异常让 CPU 在正常执行指令时，能够停下来处理外部请求或当前指令引发的问题，并在合适情况下回到原程序继续执行。

## 基本概念

CPU 正常情况下按下面循环运行：

```text
取指令 -> 执行指令 -> 取下一条指令 -> 执行下一条指令 -> ...
```

但现实中会出现两类“打断”：

1. 外设或外部事件需要 CPU 处理，例如键盘输入、定时器到时。
2. 当前指令执行不下去或需要操作系统介入，例如缺页、除零、非法指令、系统调用。

这就是中断和异常机制存在的原因。

### 中断和异常的区别

| 类型 | 来源 | 和当前指令的关系 | 典型例子 | 处理时机 |
| --- | --- | --- | --- | --- |
| 外部中断 | 外部设备或外部事件 | 通常不是当前指令造成的 | I/O 中断、定时器中断 | 通常在当前指令执行完后响应 |
| 内部异常 | 当前指令执行过程中产生 | 通常由当前指令引起 | 缺页、除零、非法指令、溢出 | 需要立即处理 |

简单记法：

```text
中断：外面有事来找 CPU
异常：CPU 执行当前指令时自己发现不对
```

### 异常的常见分类

| 类型 | 含义 | 是否可能恢复 |
| --- | --- | --- |
| 故障 Fault | 当前指令还没真正完成，遇到可处理问题 | 有些可恢复，如缺页 |
| 自陷 Trap | 程序主动触发，用于进入操作系统 | 通常处理后继续 |
| 终止 Abort | 严重硬件或系统错误 | 通常不可恢复 |

例如缺页异常属于典型 Fault：当前指令需要访问的页面不在主存中，操作系统把页面调入后，可以重新执行这条指令。

### 关键寄存器

| 名称 | 作用 |
| --- | --- |
| EPC | 保存异常或中断相关的返回地址 |
| Cause / 中断原因寄存器 | 记录发生了什么异常或中断 |
| IE / 中断允许位 | 表示当前是否允许响应中断 |
| PC | 被改写为中断或异常处理程序入口地址 |

第六章题目常问“CPU 为支持中断要增加什么”。核心就是：要能保存断点、识别原因、禁止/允许中断、跳到服务程序、再返回。

## 工作原理或推导过程

### 1. 中断处理为什么要保存现场

CPU 处理中断时会跳到另一段程序，即中断服务程序。问题是：原程序还没结束，回来时必须知道：

- 原来执行到哪里？
- 原来寄存器里保存了什么？
- 原来的状态位是什么？

否则服务程序一运行，可能把原程序的寄存器内容覆盖掉，原程序就无法正确继续。

因此中断异常处理的核心动作是：

```text
保存断点 + 保存必要现场 + 跳转处理 + 恢复现场 + 返回
```

### 2. 外部中断的一般流程

外部中断通常在当前指令执行完后响应，流程可理解为：

1. CPU 检测到中断请求，并确认当前允许中断。
2. 关中断，避免保存现场时又被打断。
3. 保存断点到 EPC。外部中断一般保存“下一条将要执行的指令地址”。
4. 记录中断原因或识别中断源。
5. 把中断服务程序入口地址送入 PC。
6. 执行中断服务程序，必要时保存和恢复通用寄存器等现场。
7. 执行返回指令，如 `eret`，把 EPC 恢复到 PC。
8. 开中断，回到原程序继续执行。

可以写成：

```text
检测中断 -> 关中断 -> 保存断点 -> 识别中断源
-> PC 改为服务程序入口 -> 服务程序处理
-> 恢复现场 -> eret 返回
```

### 3. 异常处理的断点为什么更讲究

外部中断通常和当前指令无关，所以当前指令已经执行完，返回时执行下一条指令即可。

但异常往往由当前指令引起，断点保存要看异常类型：

| 情况 | EPC 应保存什么 | 原因 |
| --- | --- | --- |
| 外部中断 | 下一条指令地址 | 当前指令已完成 |
| 缺页等可恢复故障 | 当前异常指令地址 | 页面调入后要重新执行该指令 |
| 自陷/系统调用 | 通常是返回到后续指令 | 自陷指令本身是有意执行的 |
| 严重终止异常 | 可能不返回 | 程序或系统无法继续 |

这也是考试常考点：缺页处理后通常不是执行下一条，而是重新执行导致缺页的那条指令。

### 4. CPU 数据通路要增加什么

普通 CPU 只会取指和执行指令。为了支持中断异常，还需要额外能力：

| 需求 | 可能增加的硬件 |
| --- | --- |
| 保存返回地址 | EPC 寄存器或栈机制 |
| 记录异常原因 | Cause 寄存器或中断号寄存器 |
| 临时屏蔽中断 | IE 位、开关中断控制逻辑 |
| 跳到服务程序入口 | PC 输入端增加选择器，中断入口地址逻辑 |
| 返回原程序 | `eret` 指令支持，`EPC -> PC` 数据通路 |

从数据通路角度看，中断异常就是给 PC 增加新的来源：除了 `PC+4`、分支目标地址外，PC 还可能来自“中断服务程序入口”或 `EPC`。

### 5. 微程序控制器如何支持中断

如果 CPU 用微程序控制器实现，那么中断响应也可以是一段微程序。

取指和执行结束后，微程序控制器检查是否有中断请求：

- 没有中断：微地址回到取指微程序入口。
- 有中断：微地址转到中断响应微程序入口。

中断响应微程序负责完成关中断、保存 EPC、设置 PC 为中断入口等动作。

## 典型例子

### 例 1：缺页异常

某条 `lw` 指令要访问虚拟地址对应的页面，但该页面不在主存中。

处理过程：

1. CPU 或存储管理部件发现缺页。
2. 保存当前 `lw` 指令地址到 EPC。
3. 进入操作系统缺页处理程序。
4. 操作系统把所需页面从外存调入主存，并修改页表。
5. 执行异常返回。
6. CPU 重新执行这条 `lw`。

为什么要重新执行？因为第一次 `lw` 并没有真正拿到数据，不能直接跳到下一条。

### 例 2：定时器中断

操作系统用定时器中断实现进程调度。假设当前用户程序正在运行，定时器到时：

1. 当前指令执行完。
2. CPU 保存下一条指令地址到 EPC。
3. 跳到定时器中断服务程序。
4. 操作系统决定是否切换进程。
5. 若继续当前进程，则恢复现场并返回 EPC。

这里外部中断和当前指令本身无关，所以通常不需要重新执行当前指令。

## 常见考法

- 区分外部中断、内部异常、故障、自陷、终止。
- 说明中断处理的一般流程。
- 判断 EPC 应保存当前指令地址还是下一条指令地址。
- 分析 CPU 为支持中断异常需要增加哪些寄存器和数据通路。
- 说明为什么处理中断前要关中断。
- 比较中断现场保护和函数调用保存寄存器的区别。
- 判断缺页异常处理后应重新执行哪条指令。

## 易错点

- 中断不一定是错误，很多中断是正常 I/O 或操作系统调度机制。
- 异常通常和当前指令有关，外部中断通常和当前指令无关。
- 外部中断一般在当前指令执行完后处理，缺页等异常可能要求当前指令重新执行。
- 关中断不是永远禁止中断，而是保护关键处理过程不被再次打断。
- EPC 保存的是“返回所需地址”，不总是简单等于 `PC+4`。
- 保存断点和保护现场不是一回事：断点主要是 PC，现场还包括寄存器和状态字等。
- 中断服务程序也是程序，但它由硬件或操作系统机制跳转进入，不是普通顺序调用。

## 小结

- 中断和异常让 CPU 能处理外部事件和当前指令引发的问题。
- 外部中断通常处理完当前指令后响应；异常通常与当前指令有关。
- 处理流程核心是：检测、关中断、保存断点、识别原因、转入处理程序、恢复现场、返回。
- 支持中断异常需要 EPC、Cause/中断号、IE 位、中断入口选择逻辑和 `eret` 等机制。
- 判断返回地址时要看事件类型：外部中断多返回下一条，缺页等故障多重新执行当前指令。
