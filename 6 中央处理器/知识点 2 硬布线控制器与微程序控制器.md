# 硬布线控制器与微程序控制器

> 来源依据：第 6 章 PPT“硬布线控制器”“微程序控制器工作原理”“微指令格式”“硬布线与微程序对比”。

## 一句话理解

硬布线控制器是“用电路直接算出控制信号”，微程序控制器是“把控制信号序列像程序一样存起来，再一条条取出来执行”。

## 基本概念

CPU 执行指令时，真正让硬件动作的是控制信号。例如：

```text
PCout, ARin, Read, DRout, IRin, RegWrite, ALUSrc, MemWrite
```

这些信号本质上是在问：

- 哪个寄存器允许输出？
- 哪个寄存器允许写入？
- ALU 做加法、减法还是比较？
- 存储器读还是写？
- 下一条控制信号从哪里来？

控制器的任务就是在正确的时刻产生正确的控制信号序列。

### 常见术语

| 术语 | 含义 | 直觉 |
| --- | --- | --- |
| 微操作 | CPU 内部一步最基本的动作 | `PC -> MAR`、`R[rs] -> X` |
| 微命令 | 控制某个微操作的信号 | `PCout`、`MARin` |
| 微指令 | 一个时钟周期内要发出的微命令集合，加上下条地址信息 | 一行“硬件动作清单” |
| 微程序 | 实现一条机器指令的一串微指令 | 一小段“硬件动作脚本” |
| 控制存储器 | 存放微程序的存储器 | CPU 内部的微程序仓库 |

注意层次关系：

```text
程序由机器指令组成
机器指令由微程序实现
微程序由微指令组成
微指令发出微命令
微命令控制微操作
```

## 工作原理或推导过程

### 1. 硬布线控制器怎么工作

硬布线控制器不存“控制程序”，而是用组合逻辑和状态机直接产生控制信号。

可以抽象成：

```text
指令译码信号 + 当前状态 + 反馈条件
        -> 组合逻辑电路
        -> 控制信号 + 下一状态
```

例如，当前状态是取指阶段 T1，那么不管是什么指令，都可能产生：

```text
PCout = 1
ARin = 1
Xin = 1
```

当取指和译码结束后，控制器根据 `lw`、`sw`、`beq`、`add` 等译码结果进入不同状态路径。

硬布线设计的一般步骤是：

1. 分析数据通路，明确每条指令每一步需要哪些微操作。
2. 画指令周期流程图或状态转换图。
3. 写状态转换表。
4. 对每个控制信号找出它在什么条件下为 1。
5. 写出逻辑表达式并用门电路实现。

它快的原因是：控制信号由电路直接产生，不需要先到控制存储器里取微指令。它难改的原因也在这里：指令功能变了，电路逻辑可能就要重设计。

### 2. 微程序控制器怎么工作

微程序控制器把“控制信号序列”存进控制存储器。执行机器指令时，CPU 先找到对应微程序入口，然后逐条取微指令。

基本流程：

```text
机器指令译码
-> 找到微程序入口地址
-> 取出一条微指令
-> 微指令的操作控制字段发出控制信号
-> 顺序控制字段决定下一条微指令地址
-> 重复直到该机器指令完成
```

一条微指令通常包含两个主要部分：

| 字段 | 作用 |
| --- | --- |
| 操作控制字段 | 指出本周期哪些微命令有效 |
| 顺序控制字段 | 指出下一条微指令地址如何产生 |

如果一条 `lw` 需要 5 个时钟周期完成，那么它通常至少需要若干条微指令分别完成“取指、算地址、访存、写回”等动作。课件中特别强调：**一条微指令对应一个时钟周期**。

### 3. 为什么控制存储器不是主存

控制存储器和主存都叫“存储器”，但用途完全不同：

| 项目 | 主存 | 控制存储器 |
| --- | --- | --- |
| 存放内容 | 普通程序和数据 | 微程序 |
| 使用者 | CPU 执行普通机器指令时访问 | 控制器内部访问 |
| 容量 | 通常较大 | 通常较小 |
| 字长 | 与机器字长、数据宽度相关 | 与微指令长度相关 |
| 速度 | 相对较慢 | 要很快 |
| 程序员是否直接可见 | 可通过程序访问 | 通常不可见 |

所以不能把“主存中的程序”和“控制存储器中的微程序”混为一谈。

### 4. 微指令编码方式

微指令要表达很多控制信号。如果每个控制信号用 1 位，最直观，但字长很长。

| 编码方式 | 做法 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 直接表示法 | 一个微命令占一位 | 简单、并行性强、译码快 | 微指令很长，控制存储器容量大 |
| 编码表示法 | 对互斥微命令分组编码 | 字长较短 | 需要译码，速度可能下降 |
| 混合表示法 | 直接和编码结合 | 折中 | 设计更复杂 |

判断能否编码的关键是“互斥”。例如同一总线上多个输出信号通常互斥，因为同一时刻不能让多个部件同时输出到总线；这类信号适合编码压缩。

若某组有 `m` 个互斥微命令，并且还需要表示“无操作”，则至少需要：

```text
ceil(log2(m + 1))
```

位来编码。

### 5. 水平型和垂直型微指令

| 类型 | 特点 | 直觉 |
| --- | --- | --- |
| 水平型微指令 | 一条微指令可同时发出多个微命令，字长较长 | 一行能安排很多并行动作 |
| 垂直型微指令 | 更像普通机器指令，字段短，需要译码 | 一行只表达较抽象的一类动作 |

水平型并行能力强、速度快，但控制存储器容量较大。垂直型字长短、容量小，但执行时要译码，速度较慢。

### 6. 硬布线和微程序对比

| 比较项 | 硬布线控制器 | 微程序控制器 |
| --- | --- | --- |
| 本质 | 用逻辑电路直接产生控制信号 | 从控制存储器取微指令产生控制信号 |
| 速度 | 快 | 相对慢 |
| 设计复杂度 | 电路复杂，调试困难 | 思路清楚，像写控制程序 |
| 修改灵活性 | 修改困难 | 修改或扩展较方便 |
| 成本 | 复杂时成本较高 | 可降低复杂控制逻辑设计难度 |
| 适合场景 | RISC、规则指令、高速处理器 | CISC、复杂指令、兼容性要求高 |

一句话记忆：

```text
硬布线：快、硬、难改
微程序：慢一些、软一些、好改
```

## 典型例子

### 例 1：取指微程序怎么理解

假设取指过程分为：

```text
T1: PC -> AR, PC -> X
T2: X + 4 -> Z
T3: Z -> PC, M[AR] -> DR
T4: DR -> IR
```

用微程序实现时，可以把每个节拍变成一条微指令：

| 微指令 | 操作控制字段表达的动作 |
| --- | --- |
| u1 | `PCout, ARin, Xin` |
| u2 | `+4` |
| u3 | `Zout, PCin, Read, DREin` |
| u4 | `DRout, IRin`，并根据指令译码跳到对应微程序入口 |

这样做的意义是：控制信号序列不再由复杂电路硬推出来，而是像表格一样存在控制存储器中。

### 例 2：控制存储器容量计算

若控制存储器中有 `128` 条微指令，每条微指令 `32` 位，则容量为：

```text
128 * 32 = 4096 bit
```

微地址至少需要：

```text
log2(128) = 7 bit
```

如果题目给“微命令数、判别字段、下址字段”等信息，要先算出一条微指令的字长，再乘微指令条数。

## 常见考法

- 比较硬布线控制器和微程序控制器的速度、灵活性和适用场景。
- 区分微操作、微命令、微指令、微程序、机器指令。
- 根据微命令数量、下址字段位数等计算微指令字长。
- 根据控制存储器容量和微指令字长计算微指令条数或微地址位数。
- 判断哪些微命令相容、哪些互斥，能否编码压缩。
- 说明微程序入口地址如何由机器指令译码得到。

## 易错点

- 微指令不是机器指令，微程序也不是普通应用程序。
- 控制存储器存的是微程序，不是普通用户程序。
- 一条机器指令通常对应多条微指令。
- 直接表示法不是“低级”，它速度快、并行性强，只是字长长。
- 编码表示法能缩短字长，但需要译码，不一定更快。
- 硬布线速度快，但不是“简单”；微程序容易设计，但不是“没有硬件”。
- 互斥微命令才能放心编码，同一周期可能同时有效的微命令不能放进同一个互斥编码字段。

## 小结

- 控制器的本质任务是产生正确的控制信号序列。
- 硬布线控制器用组合逻辑和状态机直接产生控制信号，速度快但难改。
- 微程序控制器把控制信号序列存成微指令，结构清晰且易修改，但速度相对慢。
- 微指令通常由操作控制字段和顺序控制字段组成。
- 做计算题时先确定微指令字长，再计算控制存储器容量或微地址位数。
