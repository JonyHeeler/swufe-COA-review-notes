# 总线仲裁、定时与系统结构

> 来源依据：第 8 章 PPT“总线的仲裁”“集中式仲裁”“总线定时”“总线连接方式”“单总线结构”“双总线结构”“三总线结构”“总线结构与系统性能关系”。

## 一句话理解

总线仲裁解决“谁先用总线”，总线定时解决“什么时候发和什么时候收”，总线结构解决“CPU、主存和 I/O 设备怎样连接才不容易堵车”。

## 基本概念

总线是共享资源。共享带来三个必须解决的问题：

```text
多个设备都想用：谁先用？
数据在线上传输：双方怎么知道数据有效？
系统部件很多：怎样连接才不会所有通信都挤在一条线上？
```

对应三个知识点：

- **总线仲裁**：分配总线使用权，避免多个主设备同时驱动总线。
- **总线定时**：规定通信双方如何确认传输开始、数据有效和传输结束。
- **总线结构**：规定系统中不同层次总线和部件的连接方式。

主设备要使用总线时，会向总线控制器发出请求。控制器按某种规则授权其中一个主设备。获得授权的主设备才能成为当前总线主控者。

## 工作原理或推导过程

### 1. 为什么需要总线仲裁

如果 CPU 和 DMA 控制器同时想在数据总线上输出数据，一根线不可能同时表示 0 和 1。轻则数据错误，重则电路冲突。因此必须先决定谁能使用总线。

仲裁的基本过程是：

```text
设备请求总线 -> 仲裁器判断优先级 -> 授权某个设备 -> 设备使用总线 -> 设备释放总线
```

按照总线控制器的位置，PPT 将仲裁分为：

- **集中式仲裁**：有一个中央仲裁器统一分配总线。
- **分布式仲裁**：没有单一中央仲裁器，各设备通过约定规则共同决定。

本章重点是集中式仲裁。

### 2. 链式查询方式：授权信号一路传下去

链式查询可以想成“排成一条队，授权牌从队头往后传”。中央仲裁器发出总线授权信号 BG，信号依次经过各个设备接口。谁正在请求总线，谁就截获授权，成为总线主设备。

PPT 中给出的控制线：

```text
BS：总线状态
BR：总线请求
BG：总线授权
共 3 根控制线
```

仲裁过程：

1. 仲裁器监控 BS，确认总线是否空闲。
2. 某设备通过 BR 表示有总线请求。
3. 仲裁器发出 BG。
4. BG 沿链路依次传递。
5. 最先遇到且正在请求的设备获得授权。
6. 设备使用总线时置 BS，使用完后释放。

优点：

- 控制线少，硬件简单。
- 扩展设备比较容易。

缺点：

- 优先级固定，离仲裁器越近优先级越高。
- 低优先级设备可能长期得不到总线，出现饥饿。
- 响应慢，因为 BG 要逐级传递。
- 对链路故障敏感，前面设备或连线出故障可能影响后面设备。

记忆方法：链式查询的关键词是“线少、固定优先级、慢、故障敏感”。

### 3. 计数器定时查询方式：仲裁器按地址逐个点名

计数器定时查询不再让授权信号沿设备链传下去，而是仲裁器通过计数线输出一个设备编号，像老师按学号点名：

```text
现在查询 0 号设备，有请求吗？
现在查询 1 号设备，有请求吗？
现在查询 2 号设备，有请求吗？
```

PPT 中给出的控制线数量：

```text
BS + BR + log2 n 根设备地址计数线
共 2 + log2 n 根控制线
```

其中 `n` 是设备数。为什么需要 `log2 n` 根地址线？因为 `m` 根二进制线最多能表示 `2^m` 个编号。要表示 `n` 个设备编号，就需要：

```text
m >= log2 n
```

所以地址计数线约为 `log2 n` 根。

优点：

- 优先级可以改变，例如计数器从不同编号开始，就能调整谁先被查询。
- 不像链式查询那样依赖授权信号逐级传递，故障不那么敏感。

缺点：

- 仍然要按编号查询，响应速度不算快。
- 增加设备时，编号范围和计数线设计可能要调整，扩展较困难。

记忆方法：计数器定时查询的关键词是“点名、线数中等、优先级可变、扩展难”。

### 4. 独立请求方式：每个设备都有自己的请求线和授权线

独立请求方式最直接：

```text
每个设备单独向仲裁器请求，仲裁器也单独给每个设备授权。
```

PPT 中给出的控制线数量：

```text
n 根 BR 请求线 + n 根 BG 授权线 + 1 根 BS 总线状态线
共 2n + 1 根控制线
```

优点：

- 响应快，因为仲裁器同时看到所有请求。
- 优先级可灵活设置，可以固定优先级，也可以动态轮转。
- 某个设备故障一般不影响其他设备请求。
- 扩展相对容易，只需增加对应请求和授权线。

缺点：

- 控制线多，硬件复杂，成本高。

记忆方法：独立请求的关键词是“线多、快、灵活、可靠”。

### 5. 三种集中式仲裁对比

| 方式 | 控制线数量 | 响应速度 | 优先级 | 故障敏感度 | 扩展性 |
| --- | --- | --- | --- | --- | --- |
| 链式查询 | 3 根 | 慢 | 固定 | 非常敏感 | 容易 |
| 计数器定时查询 | `2 + log2 n` 根 | 慢 | 可适当变化 | 不敏感 | 较难 |
| 独立请求 | `2n + 1` 根 | 快 | 可灵活变化 | 不敏感 | 容易 |

表格不要孤立背。抓住硬件代价和响应速度的取舍：

```text
线越少，硬件越简单，但判断和授权通常越慢、越不灵活。
线越多，硬件越复杂，但响应更快、控制更灵活。
```

### 6. 总线定时：怎样确认数据有效

总线定时要解决的问题是：

```text
发送方什么时候把地址和数据放到总线上？
接收方什么时候读取？
双方怎样知道对方已经准备好？
```

PPT 中给出三类：

| 定时方式 | 核心机制 | 适用情况 | 主要问题 |
| --- | --- | --- | --- |
| 同步方式 | 用公共时钟控制传输过程 | 快速设备、短距离、速度差异小 | 受最慢设备限制 |
| 异步方式 | 用请求、应答等握手信号控制 | 慢速设备、速度差异大 | 控制复杂，握手开销较大 |
| 半同步方式 | 公共时钟加等待/应答机制 | 速度不完全一致的设备 | 比同步复杂，比异步规整 |

#### 同步方式

同步方式像按铃上课：所有设备看同一个时钟节拍。

优点：

- 控制简单。
- 高速设备之间效率高。

缺点：

- 一个周期要留给最慢设备足够时间。
- 设备速度差异大时，快设备也要等慢设备。
- 适合短距离，因为距离长会带来时钟偏斜。

#### 异步方式

异步方式像两个人递东西时互相确认：

```text
发送方：我准备好了。
接收方：我收到了。
发送方：那我撤销信号。
```

它不依赖统一时钟，而靠握手信号配合。适合慢速设备或速度差异大的设备。

PPT 提到异步方式可分为非互锁、半互锁、全互锁。互锁程度越高，双方确认越充分，可靠性越好，但过程越慢。

#### 半同步方式

半同步方式介于两者之间：基本按同步时钟工作，但允许慢设备通过等待信号延长传输。它的直觉是：

```text
大家按统一节拍走，但慢设备可以举手说“等一下”。
```

### 7. 分离事务通信方式

分离事务的思想是把一次事务拆开：

```text
先发出请求，然后释放总线；等从设备准备好数据后，再重新占用总线返回结果。
```

这样可以避免主设备在等待慢设备时长时间占着总线。它适合访问延迟较长的场景，有利于提高总线利用率，但控制更复杂。

### 8. 总线结构：从一条路到分层道路

总线结构决定 CPU、主存、I/O 设备怎样连接。它影响系统吞吐量、扩展能力、指令系统和地址空间。

#### 单总线结构

所有主要部件共享同一条系统总线。

优点：

- 结构简单。
- 使用灵活。
- 扩展容易。
- 可以统一编址，不一定需要专门 I/O 指令。

缺点：

- 所有通信都挤在同一条总线上，容易成为瓶颈。
- 高速设备发挥不出高速特性。
- 内存地址空间可能要为外设保留一部分地址。

直觉：单总线像全城只有一条主路，修起来简单，但车一多就堵。

#### 双总线结构

双总线结构把通信分开。PPT 中给出两种典型思路：

- 以主存为中心，用存储总线连接 CPU 和主存，用系统总线连接 I/O。
- 通过桥接器把高速局部总线和慢速 I/O 总线连接起来。

优点：

- 降低系统总线负载。
- 提高并行性。
- 慢速设备不会总是占用高速部件通道。

可能代价：

- 结构比单总线复杂。
- 某些结构需要专门 I/O 指令。

#### 三总线结构

三总线结构进一步分层，把高速与低速传输分离。PPT 强调：

```text
高速设备靠近 CPU，慢速设备远离 CPU，不同层次总线之间通过桥接方式连接和缓冲。
```

优点：

- CPU、主存、高速 I/O、低速 I/O 可以分层通信。
- 吞吐量比单总线结构大得多。
- 高速设备和低速设备互相干扰减少。

缺点：

- 桥接和控制更复杂。
- 成本更高。

## 典型例子

### 例 1：8 个设备用独立请求方式需要多少控制线

独立请求方式：

```text
控制线 = 2n + 1
n = 8
控制线 = 2 × 8 + 1 = 17 根
```

其中 8 根请求线、8 根授权线、1 根总线状态线。

### 例 2：16 个设备用计数器定时查询需要多少控制线

计数器定时查询：

```text
控制线 = 2 + log2 n
n = 16
log2 16 = 4
控制线 = 2 + 4 = 6 根
```

这 4 根设备地址计数线可以表示 0 到 15 共 16 个编号。

### 例 3：为什么同步方式受最慢设备限制

假设总线上有一个设备需要 10ns 才能准备好数据，另一个设备只要 2ns。同步总线如果用统一时钟，周期必须保证最慢设备也能完成。因此快设备即使 2ns 完成，也可能要等到 10ns 的节拍。

### 例 4：为什么三总线结构吞吐量更大

单总线中，CPU 访问主存、磁盘 DMA、I/O 设备传输都争同一条路。三总线结构把高速和低速设备分层，一部分通信可以在不同层次并行进行，所以系统吞吐量更高。

## 常见考法

- 比较链式查询、计数器定时查询、独立请求的控制线数量和特点。
- 根据设备数计算仲裁控制线数量。
- 判断哪种仲裁方式响应快、优先级灵活或硬件简单。
- 比较同步、异步、半同步定时方式的适用场景。
- 解释为什么同步总线受最慢设备限制。
- 比较单总线、双总线、三总线结构的优缺点。
- 说明总线结构对最大存储容量、指令系统和吞吐量的影响。

## 易错点

- **认为链式查询优先级可随意改变**：链式查询优先级由设备离仲裁器的远近决定，通常固定。
- **把 BR 和 BG 混淆**：BR 是设备向仲裁器请求，BG 是仲裁器向设备授权。
- **忘记 BS 总线状态线**：计算控制线数量时，独立请求是 `2n + 1`，不是 `2n`。
- **认为异步方式没有控制规则**：异步不是随意传，而是靠握手信号控制。
- **认为同步方式总是最快**：速度差异大时，同步周期要照顾最慢设备。
- **把双总线、三总线只理解成线更多**：它们的关键是分离不同层次、不同速度的通信流量。
- **认为单总线一定需要专门 I/O 指令**：单总线常可统一编址，因此指令系统可能更简单。

## 小结

- 仲裁解决多个主设备争用总线的问题。
- 链式查询线少但慢且优先级固定；计数器定时查询线数中等且优先级可变；独立请求线多但快且灵活。
- 定时解决通信双方如何确认开始、有效和结束。
- 同步靠公共时钟，异步靠握手，半同步兼有两者特点。
- 单总线简单但容易堵，双总线和三总线通过分层分流提高系统吞吐量。
