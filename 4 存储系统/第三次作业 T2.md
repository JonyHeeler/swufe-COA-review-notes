# 第三次作业 第二题

## 原题

> 若计算机主存地址为 32 位，按字节编址，Cache 数据区大小为 32KB，主存块大小为 32B，采用直接映射方式和回写（Write Back）策略，则 Cache 行的位数至少是多少？

## 涉及知识点与公式

第一，来自`核心知识点 1 主存按字节编址与容量计算.md`：

按字节编址时，一个地址对应 1B，`n` 位地址最多表示 `2^n` 个地址；所以题目说“主存地址为 32 位，按字节编址”，意思是主存地址一共有 32 个二进制位，这 32 位地址要被拆成 `Tag | Index | Offset` 三部分。

第二，来自`核心知识点 2 Cache与主存映射替换写策略.md`：

Cache 地址字段的基本公式是 

> offset 位数 = log2(块大小字节数)

> Cache 行数 = Cache 数据容量 / 块大小

直接映射时可以理解为“每个主存块只能放入唯一一个 Cache 行”，所以 `index 位数 = log2(Cache 行数)`，最后 `tag 位数 = 主存地址位数 - index 位数 - offset 位数`。

此外，Cache 行中不只有数据块本身，还要有用于判断“这一行装的是哪一个主存块”的 Tag、用于判断内容是否有效的有效位 valid bit；如果采用回写 Write Back 策略，还必须有脏位 dirty bit，因为 CPU 写 Cache 后主存可能还没同步更新，系统需要靠脏位判断替换时是否要写回主存。

因此本题中的 Cache 行位数至少应按：`Cache 行位数 = 数据块位数 + Tag 位数 + 有效位 + 脏位` 计算。

## 解题思路与答案

Cache 数据区大小为 32KB，注意这里说的是“数据区”，也就是只算真正存放数据的部分，不包括 Tag、有效位、脏位等额外控制位；

每个主存块大小为 32B，所以 Cache 一共能放 `32KB / 32B = 2^15B / 2^5B = 2^10 = 1024` 个主存块，也就是有 1024 个 Cache 行。

采用直接映射时，每个主存块只能对应到一个固定 Cache 行，CPU 访问时要先用地址中的 Index 找到那一行；既然共有 1024 行，就需要 `log2(1024)=10` 位 Index 来区分这些行。

主存地址总长是 32 位，其中低 5 位已经用来表示块内偏移，中间 10 位已经用来表示 Cache 行号，剩下的高位就是 Tag，所以 `Tag 位数 = 32 - 10 - 5 = 17` 位。

到这里为止，我们只是把主存地址拆成了 `Tag 17 位 | Index 10 位 | Offset 5 位`。

但是题目问“Cache 行的位数”，Cache 行里面并不会存 Index 和 Offset：Index 是 CPU 拿来“找到这一行”的地址字段，不需要再存进这一行；Offset 是 CPU 在已经命中的数据块内部“选第几个字节”的字段，也不需要作为每行标签保存。真正要存进每个 Cache 行的是：数据块本身 `32B = 32 × 8 = 256` 位，Tag `17` 位，有效位 `1` 位；又因为题目明确采用回写 Write Back 策略，所以还要有脏位 `1` 位。

于是每个 Cache 行至少需要 `256 + 17 + 1 + 1 = 275` 位。答案：**Cache 行的位数至少是 275 位**。

也可以把整个过程压缩成一行公式：

`块内偏移 = log2(32B)=5`，`Cache 行数 = 32KB/32B = 1024 = 2^10`，`Index=10`，`Tag=32-10-5=17`，`Cache 行位数 = 数据 32B×8 + Tag 17 + 有效位 1 + 脏位 1 = 256+17+1+1=275 位`。

## 拓展设问与变种考法

> **变式 1：如果题目把“回写 Write Back”改成“写穿 Write Through”，Cache 行至少多少位？** 

写穿法通常不需要脏位，因为每次写 Cache 时会同时写主存，主存始终保持最新，不需要等替换时再判断是否写回。其他字段不变，数据块仍为 256 位，Tag 仍为 17 位，有效位仍为 1 位，所以 Cache 行位数为 `256 + 17 + 1 = 274` 位。

> **变式 2：如果题目说“不考虑有效位和脏位”，Cache 行至少多少位？**

 这类题是在故意考你有没有看清“是否考虑控制位”。如果不考虑有效位、脏位，那么每行只算数据块和 Tag，结果是 `256 + 17 = 273` 位。但常规题目问“至少多少位”并且给了回写策略，一般应把有效位和脏位都算上。

> **变式 3：若 Cache 数据区仍为 32KB，但主存块大小改为 64B，仍然直接映射、回写，Cache 行至少多少位？** 

块变大后，数据字段变为 `64B×8=512` 位，Offset 变为 `log2(64)=6` 位；Cache 行数变为 `32KB/64B = 2^15/2^6 = 2^9 = 512` 行，所以 Index 为 9 位；Tag 为 `32-9-6=17` 位。回写仍要有效位和脏位，因此行位数为 `512 + 17 + 1 + 1 = 531` 位。这里很容易发现：块从 32B 变成 64B 后，Tag 没变，但数据字段直接翻倍，所以行位数明显变大。

> **变式 4：若 Cache 数据区改为 16KB，主存块仍为 32B，直接映射、回写，Cache 行至少多少位？**

Cache 行数变为 `16KB/32B = 2^14/2^5 = 2^9 = 512` 行，所以 Index 为 9 位；Offset 仍为 5 位；Tag 为 `32-9-5=18` 位。数据字段仍为 256 位，加有效位和脏位后，Cache 行位数为 `256 + 18 + 1 + 1 = 276` 位。注意 Cache 变小后，Index 少了 1 位，Tag 反而多了 1 位，因为主存地址总位数不变，行号位数少了，高位 Tag 就要多承担一位身份区分。

> **变式 5：若主存地址改为 36 位，其他条件仍为 32KB Cache、32B 块、直接映射、回写，Cache 行至少多少位？**

 Offset 仍为 5 位，Cache 行数仍为 1024 行，所以 Index 仍为 10 位；只有主存地址总位数从 32 位变为 36 位，因此 Tag 变为 `36-10-5=21` 位。每行位数为 `256 + 21 + 1 + 1 = 279` 位。主存地址位数越长，说明主存可寻址空间越大，在 Cache 行数和块大小不变时，会有更多主存块竞争同一个 Cache 行，所以需要更长的 Tag 来区分。

> **变式 6：若映射方式改为二路组相联，其他条件仍为 32 位地址、32KB Cache、32B 块、回写，每个 Cache 行至少多少位？**

先算总行数仍是 `32KB/32B=1024` 行，但二路组相联表示每组 2 行，所以组数是 `1024/2=512=2^9` 组，Index 变为 9 位；Offset 仍是 5 位；Tag 变为 `32-9-5=18` 位。每一行仍然只放一个 32B 数据块，因此数据字段还是 256 位，再加 Tag、有效位、脏位，结果是 `256 + 18 + 1 + 1 = 276` 位。这里要注意：二路组相联改变的是“组号位数”和“Tag 位数”，不是让一个 Cache 行放两个主存块。

> **变式 7：若问原题的地址字段如何划分？** 

原题中 Offset 为 5 位，Index 为 10 位，Tag 为 17 位，所以地址字段划分为 `Tag 17 位 | Index 10 位 | Offset 5 位`。这类题只问地址字段，不是问 Cache 行位数，因此不要把数据块 256 位、有效位、脏位加进去。

> **变式 8：若问原题中主存块号 129 应映射到哪一个 Cache 行？** 

直接映射的行号公式是 `Cache 行号 = 主存块号 mod Cache 行数`。原题 Cache 行数是 1024，所以 `129 mod 1024 = 129`，应映射到第 129 行。如果主存块号是 1153，那么 `1153 mod 1024 = 129`，也映射到第 129 行，因此这两个主存块会发生冲突，不能同时留在直接映射 Cache 的同一行中。

> **变式 9：若问原题中主存地址 129 号单元映射到哪一个 Cache 行？** 

这和“主存块号 129”不是一回事。主存地址 129 是字节地址，先要除以块大小 32B，得到它所在的主存块号：`129 / 32 = 4 余 1`，所以它属于主存第 4 块，块内偏移为 1。直接映射行号为 `4 mod 1024 = 4`，所以装入第 4 行。这个变式专门考“字节地址”和“主存块号”不要混淆。

> **变式 10：若问原题的 Cache 总位数至少是多少？** 

原题每行至少 275 位，一共有 1024 行，所以总位数至少为 `275 × 1024 = 281600` 位，也就是 `281600 / 8 = 35200B`。注意这个数大于 32KB，因为 32KB 只是 Cache 数据区大小，不包括 Tag、有效位、脏位这些附加位。

> **变式 11：若问每个 Cache 行为什么不保存 Index 和 Offset？** 

Index 的作用是“从很多 Cache 行中找到某一行”，它是访问时由地址直接送到 Cache 的选择信号，相当于你已经按楼层和房间号走到了某个房间门口，不需要在房间里面再存一份“我是几号房”。

Offset 的作用是在一个数据块内部选择具体字节，比如 32B 的块内部选第 0 到第 31 个字节，它也由 CPU 地址的低位直接给出，不需要作为每行的额外管理信息保存。

Cache 行必须保存 Tag，是因为多个不同主存块可能映射到同一行，CPU 走到这一行后还必须核对“房间里住的是不是我要找的人”。

> **变式 12：若问为什么回写策略一定要有脏位？** 

因为回写法下 CPU 写数据时通常只修改 Cache，不马上改主存。这样一来，某一行 Cache 可能出现两种状态：一种是“没有被改过”，替换时直接丢掉即可；另一种是“被改过但还没写回主存”，替换前必须先把它写回主存。

脏位就是用来区分这两种状态的 1 位标志，所以原题使用 Write Back 时，每个 Cache 行至少要加 1 位 Dirty Bit。

## 最后小结

这类题的核心套路是：先拆地址，再算每行附加信息。地址拆分看 `Tag | Index | Offset`：Offset 看块大小，Index 看 Cache 行数或组数，Tag 是主存地址剩下的高位；Cache 行位数看“这一行里面实际要保存什么”：数据块一定要保存，Tag 一定要保存，有效位通常要保存，Write Back 时还要保存脏位。
