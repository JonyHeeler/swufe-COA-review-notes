# 虚拟存储器与虚实地址转换

> 来源依据：`知识点汇总.txt` 第四章第 3 点；第 4 章 PPT“虚拟存储器”“虚实地址转化”；教材第 4 章。

## 一句话理解

虚拟存储器让程序以为自己拥有一大片连续空间，真正运行时再由硬件和操作系统把“程序看到的地址”翻译成“主存里的真实地址”。

## 基本概念

普通程序里看到的地址通常不是物理内存的真实地址，而是**虚拟地址**。

- **虚拟地址 VA**：程序发出的地址，也叫逻辑地址。
- **物理地址 PA**：主存硬件真正使用的地址，也叫实地址。
- **虚拟页**：操作系统把程序看到的那个“虚拟的连续大空间”，像切蛋糕一样，切成一个个固定大小的小块（通常是 4KB）。这些小块就叫**虚拟页**。
- **VPN (虚拟页号)**：每个虚拟页都有一个编号，相当于这块“蛋糕”的序号。
- **物理页框**：真正插在电脑主板上的物理内存条，也会被切成和虚拟页**完全一样大小**的小块。这些真实存在的小块被称为**物理页框**。
- **PPN (物理页号)**：真实内存中小块的编号。
- **页表**：因为程序的虚拟页实际上被打散，乱七八糟地塞在各个物理页框里（有的甚至还没放进内存，被放在了硬盘上），所以必须有一个“大账本”来记录这种对应关系。

- **页表项 PTE**：页表中的一条记录，通常包括有效位、物理页号、权限位、脏位等。**有效位**：这个虚拟页现在到底在不在物理内存里？（如果在，就是 1；如果还在硬盘上没搬过来，就是 0，这时候 CPU 就会触发“缺页异常”去硬盘里捞它）。**权限位**：这个页面是只读的，还是可读可写的？**脏位**：这个页面里的数据有没有被修改过？

- **TLB**：页表项的高速缓存，用来加快地址翻译。

可以把虚拟地址理解成“学生手里的座位号”，物理地址理解成“教室里真正的座位”。页表就是座位安排表：学生编号对应哪个真实座位。TLB 是老师手边的小抄，记录最近查过的几条座位安排，避免每次都翻完整表。

虚拟存储器解决两个问题：

- **空间管理**：每个程序都像拥有独立、连续的地址空间。
- **容量扩展**：暂时不用的页可以放在外存，需要时再调入主存。

## 工作原理或推导过程

### 1. 为什么页内偏移不变

分页时，虚拟空间和物理空间使用相同的页面大小。

例如页面大小为 4KB，那么每一页内部都有：

```text
0 ~ 4095
```

这些页内位置。

地址翻译只改变“第几页”，不改变“这一页内部第几个字节”。所以：

```text
虚拟地址 = 虚拟页号 VPN | 页内偏移 VPO
物理地址 = 物理页号 PPN | 页内偏移 PPO
```

并且：

```text
VPO = PPO
```

这点非常重要。做题时只要页面大小确定，虚拟地址和物理地址的低位偏移部分就已经确定。

### 2. 字段位数怎么求

页面大小为 `2^p` 字节时，页内偏移需要 `p` 位。

```text
VPO 位数 = PPO 位数 = p
VPN 位数 = 虚拟地址位数 - p
PPN 位数 = 物理地址位数 - p
```

例如页面大小 4KB：

```text
4KB = 2^12B
```

所以页内偏移为 12 位。

### 3. 虚实地址转换流程

CPU 发出虚拟地址后，转换过程可以按下面顺序理解：

1. 从虚拟地址中拆出 `VPN` 和 `VPO`。
2. 用 `VPN` 查 TLB。
3. TLB 命中：直接得到对应的 `PPN`。
4. TLB 未命中：去主存中的页表查对应页表项。
5. 页表项有效：得到 `PPN`，并可能把这条映射填入 TLB。
6. 页表项无效：发生缺页异常，由操作系统把页面从外存调入主存。
7. 用 `PPN` 拼接原来的页内偏移，得到物理地址。

最终拼接方式：

```text
物理地址 PA = PPN | VPO
```

这里的 `|` 表示拼接，不是按位或运算。

你可以把它想象成一个快递分发系统：程序只知道包裹的“逻辑编号”（虚拟地址），而硬件和操作系统必须配合起来，一层层闯关，最终找到仓库里真实的“物理货架号”（物理地址）。

为了方便理解，我们可以把这 7 个步骤划分为 **四个核心阶段**：

#### 阶段一：拆解准备（第 1 步）

- **动作**：CPU 拿到完整的虚拟地址后，第一件事就是把它“拦腰截断”，拆分为 **VPN（虚拟页号）** 和 **VPO（页内偏移）**。

- **意义**：这就像把地址“北京市朝阳区某小区101室”拆成“小区名”（VPN）和“具体门牌号”（VPO）。接下来，硬件只拿“小区名”去查地图，而“具体门牌号”则被先放在一边，保留备用。

#### 阶段二：高速通道（第 2、3 步）

- **动作**：拿着 VPN 去查 **TLB（快表）**。

- **意义**：TLB 是 CPU 内部的一张“高频访问小抄”，速度极快。如果运气好，这个 VPN 最近刚被查过（**TLB 命中**），小抄上直接写着对应的 PPN（物理页号），那么恭喜，直接跳到最后一步。这是最理想、最快的情况。

#### 阶段三：慢速兜底（第 4、5 步）

- **动作**：如果小抄上没有（**TLB 未命中**），CPU 只能老老实实去内存里翻那本厚厚的“总账本”——**页表**。

- **意义**：在页表里找到对应的页表项后，只要它的“有效位”为 1，说明数据确实在内存里。拿到 PPN 后，系统为了下次能查得更快，通常还会顺手**把这条记录抄写到 TLB 里**更新小抄。

#### 阶段四：异常抢救（第 6 步）

- **动作**：如果连页表里都显示“无效”，说明你要找的这个页面目前根本不在物理内存中（可能还在硬盘上），这就引发了严重的 **“缺页异常 (Page Fault)”**。

- **意义**：此时，硬件会大喊“救命”，操作系统作为大管家立刻接管控制权，费力地去外存（硬盘）把缺失的页面搬进内存，更新页表，然后再让 CPU 重新执行一遍刚才的翻译流程。

#### 最终拼接（第 7 步）

- **公式解释**：`物理地址 PA = PPN | VPO` 这里的 `|` 仅仅代表“字符拼接”**。就如同我们终于查到了“小区名”对应的“真实物理坐标”（PPN），然后把一开始保留的“具体门牌号”（VPO）原封不动地粘在坐标后面。因为虚拟页和物理页框内部大小一样，所以**页内偏移是不变的（VPO 等于 PPO）。

### 4. TLB、页表、Cache 的关系

这三个东西经常一起考，但层次不同：

| 名称    | 缓存的是什么     | 解决什么问题         |
| ----- | ---------- | -------------- |
| TLB   | 页表项        | 加快虚拟地址到物理地址的翻译 |
| 页表    | 虚页到物理页框的映射 | 记录页面放在哪里       |
| Cache | 主存数据块或指令块  | 加快 CPU 访问主存数据  |

常见访问顺序可以简化为：

```text
虚拟地址 -> 查 TLB/页表得到物理地址 -> 用物理地址访问 Cache/主存
```

所以：

- TLB 命中不等于 Cache 命中。
- TLB 未命中不等于缺页。
- 缺页一定比 Cache 缺失代价大得多，因为可能要访问外存。

## 典型例子

### 例 1：求 VPN、VPO、PPN、PPO 位数

虚拟地址空间 16MB，物理地址空间 1MB，页面大小 4KB。求各字段位数。

- **第一步：求出虚拟地址和物理地址的“总长度”。**
  
  将容量化为以字节（B）为单位的 $2$ 的幂次方：
  
  - 虚拟地址空间 $16\text{MB} = 16 \times 2^{20}\text{B} = 2^4 \times 2^{20}\text{B} = 2^{24}\text{B}$。说明需要 $24$ 位虚拟地址。
  
  - 物理地址空间 $1\text{MB} = 2^{20}\text{B}$。说明需要 $20$ 位物理地址。

- **第二步：求出页内偏移（Offset）的长度。**
  
  看页面大小：$4\text{KB} = 4 \times 2^{10}\text{B} = 2^{12}\text{B}$。
  
  指数是 $12$，说明**页内偏移（VPO 和 PPO）都是 12 位**。

- **第三步：做减法求页号。**
  
  因为 `总长度 = 页号 + 偏移量`，所以直接用总长度减去偏移量：
  
  - $\text{VPN 位数} = 24 - 12 = 12\text{位}$。
  
  - $\text{PPN 位数} = 20 - 12 = 8\text{位}$。

### 例 2：根据页表求物理地址

页面大小为 4KB，虚拟地址为 `1234H`，页表显示虚拟页 `1H` 映射到物理页框 `2AH`，求物理地址。

注意：在计算机科学中，字母 **`H` 并不是数字本身的一部分，它只是一个“标签”或“单位”**。`H` 是 Hexadecimal（十六进制）的首字母缩写。

- **明确偏移量的“十六进制位数”。**
  
  页面大小 $4\text{KB}$，刚才算过偏移量是 **12 位二进制**。
  
  在计算机底层，完整的虚拟地址是一长串的 0 和 1。其中最后 12 位是不能动的偏移量（VPO），前面剩下的位是虚拟页号（VPN）。
  
  但题目给我们的虚拟地址 `1234H` 是十六进制的。因为 **4 个二进制位 = 1 个十六进制字符**，所以这 12 位的二进制偏移量，刚好对应： $12 \div 4 = 3$ **个十六进制字符**。这意味着：在这个地址的十六进制表示中，**最后 3 个字符就是绝对不能动的页内偏移（VPO）**。

- **无痛手撕虚拟地址。**
  
  拿到虚拟地址 `1234H`，从右往左数 3 位切开：
  
  - 尾巴（VPO）：`234H`
  
  - 头部（VPN）：剩下的 `1H`

- **查表换头，拼接物理地址。**
  
  - 查表：把旧头部（VPN）`1H` 扔掉，换成页表告诉你的新头部（PPN）`2AH`。
  
  - 拼接：把新头部和绝对不变的旧尾巴拼起来：`2AH` + `234H` = **`2A234H`**。

### 例 3：TLB 未命中一定缺页吗

不一定。

TLB 只是页表项的小缓存。TLB 未命中只能说明“最近没缓存这条映射”，还要去页表查：

- 页表项有效：页面在主存中，只是 TLB 没有记录。
- 页表项无效：页面不在主存中，才发生缺页异常。

## 常见考法

- 给虚拟地址位数、物理地址位数和页面大小，求 VPN/VPO/PPN/PPO 位数。
- 给虚拟地址和页表/TLB 内容，求物理地址。
- 判断某次访问是 TLB 命中、页表命中、缺页还是 Cache 命中。
- 比较 TLB 未命中、Cache 缺失、缺页异常的处理代价。
- 判断页内偏移是否改变。

## 易错点

- **页内偏移在虚拟地址和物理地址中相同**。
- **TLB 未命中不等于缺页**，还要查页表。
- **TLB 命中说明页表项有效**，此时不会再发生缺页。
- **缺页异常处理完后通常重新执行导致缺页的指令**。
- **虚拟页号 VPN 和物理页号 PPN 位数不一定相同**，因为虚拟地址空间和物理地址空间可以大小不同。
- **TLB 缓存页表项，Cache 缓存主存数据块**，不要混为一谈。

## 小结

- 虚拟存储器的核心是“虚拟页号到物理页框号”的映射。
- 页面大小决定页内偏移位数。
- 地址转换时只替换页号，页内偏移保持不变。
- 做题顺序：先拆地址字段，再查 TLB/页表，最后拼物理地址。
- TLB、页表、Cache 是综合题高频组合。
