# Cache 与主存映射、替换和写策略

> 来源依据：`知识点汇总.txt` 第四章第 2 点；第 4 章 PPT“高速缓冲存储器”“cache 与主存交互”；教材第 4 章。

## 一句话理解

Cache 是放在 CPU 和主存之间的“小而快”的临时仓库，考试最核心的问题是：主存中的某一块数据应该放到 Cache 的哪里，找它时又怎样判断它在不在。

## 基本概念

CPU 访问主存很慢，但程序经常反复访问最近用过的数据，或者访问相邻的数据。Cache 就利用这种局部性，把近期可能用到的数据块提前放到离 CPU 更近的地方。

先区分几个基本词：

- **主存块**：主存按固定大小切成的块，例如每块 64B。
- **Cache 行**：Cache 中能存放一个主存块的位置，也常叫 Cache 块。
- **块内偏移 offset**：一个块内部第几个字节。
- **标记 tag**：因为多个主存块可能会映射到同一个 Cache 行，所以需要一个标记（Tag）来记录当前这个 Cache 行里装的到底是主存中的哪一块。当 CPU 来找数据时，会比对这个 Tag。
- **有效位 valid bit**：这是一个只有 1 位的标志符，用来表示这一行当前的内容是否是有效的数据。开机时 Cache 是空的，有效位为 0；装入数据后，有效位变为 1。
- **脏位 dirty bit**：这个标志位并非所有 Cache 都有。它专门在使用写回法（write-back）策略时使用，用来表示这一行的数据是否被 CPU 修改过且还未写回主存。

你可以把 **主存** 想象成一个巨大的超级中药房，**Cache** 是药剂师手边的一个临时配药盘。

- 药剂师不会因为缺一味药的一克就跑去大库房拿一克，而是把装有这味药的一个标准小药盒（**主存块**）整盒拿到配药盘的某个格子（**Cache 行**）里放着。

- 在这个格子（Cache 行）里：
  
  - 药材本身就是 **数据块**。
  
  - 格子边缘贴的药名标签就是 **Tag**（防止拿错药）。
  
  - 格子上有一盏指示灯是 **有效位**（绿灯说明格子里有药，红灯说明空着）。
  
  - 格子还有一个小夹子是 **脏位**（如果药剂师往药盒里混了新药材还没同步给大库房，就把夹子立起来）。

Cache 访问有两种结果：

- **命中**：要访问的数据已经在 Cache 中。
- **缺失**：要访问的数据不在 Cache 中，需要从主存取来。

## 工作原理或推导过程

### 1. 为什么地址要拆成 tag、index、offset

一条主存地址本质上回答三个问题：

- **组索引 (index)**：告诉 CPU 应该去 Cache 的哪一行或哪一组去寻找目标数据。

- **标记 (tag)**：相当于这一行数据的“主存身份证号”。因为成千上万个不同的主存块都有可能映射到同一个 Cache 行，CPU 必须通过对比地址里的 Tag 和 Cache 行里存的 Tag 是否一致，来最终断定：“现在躺在这一行里的是不是我要的那块数据”。

- **块内偏移 (offset)**：确认命中后，根据偏移量在这个 Cache 行的数据块中精确定位到具体的那个字节。

所以组相联 Cache 中，地址通常划分为：

```text
tag | 组索引 index | 块内偏移 offset
```

三段位数的计算顺序固定：

```text
offset 位数 = log2(块大小字节数)
Cache 行数 = Cache 数据容量 / 块大小
组数 = Cache 行数 / 相联路数
index 位数 = log2(组数)
tag 位数 = 主存地址位数 - index 位数 - offset 位数
```

其中“相联路数”就是每组有几行。二路组相联表示每组 2 行，不是有 2 组。

这五个计算步骤的推导原理：

###### 1. `offset 位数 = log2(块大小字节数)`

- **原理（块内偏移）**：主存和 Cache 之间交换数据的最小单位是“块”（Block），而不是单个字节。当 CPU 想要访问某个字节时，Cache 会把包含该字节的整个块都搬过来。

- **通俗理解**：如果一个块大小是 $64\text{B}$（相当于一个箱子里装了 64 个小件），那么在这个箱子内部，你需要编号 $0$ 到 $63$ 才能精确区分每一个小件。表示 64 个不同的位置，在二进制下刚好需要 $\log_2(64) = 6$ 位。这就是 offset 负责的工作：**在已经找到的块里，精确定位到具体的那个字节。**

###### 2. `Cache 行数 = Cache 数据容量 / 块大小`

- **原理（总行数）**：这是一个最基本的除法分配问题。

- **通俗理解**：Cache 的总容量就像是一个大仓库的总面积，而每个“块”是固定大小的标准货架。仓库总面积除以单个货架的面积，得出的自然就是这个仓库一共能放下多少个货架（Cache 行/槽位）。

###### 3. `组数 = Cache 行数 / 相联路数`

- **原理（分组映射）**：在组相联映射（Set-Associative Mapping）中，Cache 行不是散养的，而是被划分成了若干个“组”。

- **通俗理解**：**“相联路数”指的是每组有几行**。如果一共有 256 行，采用 4 路组相联（每组 4 行），那么总组数就是 $256 / 4 = 64$ 组。主存中的数据块未来只能映射到某一个特定的“组”里，但可以放在该组内的任意一“行”中。

###### 4. `index 位数 = log2(组数)`

- **原理（组索引）**：CPU 拿着地址来 Cache 找数据时，第一步就是要知道“去哪个组找”。

- **通俗理解**：如果 Cache 一共有 64 个组，为了能用二进制给这 64 个组进行唯一编号（从组 0 到组 63），你需要 $\log_2(64) = 6$ 位二进制数。地址中间的这部分 `index` 位段，就是直接充当“组号指示牌”的。

###### 5. `tag 位数 = 主存地址位数 - index 位数 - offset 位数`

- **原理（身份标记）**：这是最后剩下的部分，也是用来解决“冲突”的关键。

- **通俗理解**：因为主存极其庞大，而 Cache 很小，所以主存中会有成千上万个块被映射到 Cache 的同一个“组”里。当 CPU 通过 `index` 找到某个组时，它怎么知道组里现在躺着的数据块，是不是自己想要的那个？这就需要核对 `tag`（标记）。主存地址去掉了用于找组的 `index` 和用于找字节的 `offset`，**剩下的高位部分自然就成为了这个主存块的唯一“身份证号”**。

### 2. 三种映射方式怎么理解

理解 Cache 的三种映射方式，核心在于搞明白一个问题：**主存中的某一个数据块，被搬到 Cache 时，究竟有资格放在哪些位置？**

我们可以把主存想象成一个庞大的汽车编队（主存块），把 Cache 想象成一个只有少量车位的 VIP 停车场（Cache 行）。不同的映射方式，就是不同的停车管理规定。

根据文档，这三种映射方式的特点如下：

#### 1. 直接相联映射 (Direct Mapping) —— “固定专属车位”

- **规则**：每一辆车（主存块）只能停在唯一一个固定的车位（Cache 行）。

- **计算**：`Cache 行号 = 主存块号 mod Cache 行数`。比如 Cache 有 8 个行，第 12 号主存块只能放在 $12 \pmod 8 = 4$ 号行。

- **优缺点**：找车极其快（直接去固定车位看一眼就行），硬件实现最简单。但非常容易发生冲突，比如 4 号块和 12 号块都要抢 4 号车位，即使其他车位全空着，它们也会互相把对方挤出去。

#### 2. 全相联映射 (Fully Associative) —— “随便停”

- **规则**：任何一辆车（主存块）可以停在停车场里的任意一个空车位（任意一行）。

- **优缺点**：空间利用率最高，只要停车场没满就不会发生冲突。但是找车极其痛苦，保安必须把整个停车场的所有车牌（Tag）挨个对比一遍。因此比较器极多，硬件成本高且复杂。

#### 3. 组相联映射 (Set-Associative) —— “分区停车”

- **规则**：这是前两者的折中方案。把停车场划分成几个区（组），规定一辆车只能停在某个特定的区，但在这个区内部，它可以停在任意一个车位。

- **计算**：`Cache 组号 = 主存块号 mod Cache 组数`。比如停车场分了 4 个组（每组 2 个车位），第 12 号主存块只能去 $12 \pmod 4 = 0$ 号组，但可以停在 0 号组的任意一个空位。

- **优缺点**：既降低了直接相联的冲突概率，又避免了全相联那样极其复杂的硬件比对逻辑。通常所说的“二路组相联”，就是指每一个组里面有 2 个行（车位）。

### 3. 替换算法什么时候才需要

当 CPU 发现需要的数据不在 Cache 中（发生缺失），且 Cache 中目标位置已经装满时，就需要决定“把哪个旧数据踢出去，给新数据腾地方”**，这个做决定的规则就是**替换算法。

理解替换算法，首先要明确它使用的**前提条件**。它并非每次访问 Cache 时都会被用到，只有当以下三个条件同时满足时才需要：

1. **发生 Cache 缺失**：需要的数据不在 Cache 里。

2. **目标位置已满**：新数据需要放入的那个 Cache 组（或全相联的整个 Cache）没有空闲行了。

3. **有多个候选行可供替换**：例如在组相联或全相联映射中，有多个位置可以选择。如果是直接相联映射，因为每个主存块只能放在唯一固定的位置，没得选，所以不需要复杂的替换算法。

常见的替换算法有以下几种：

- **FIFO (First In First Out，先进先出)**：最简单粗暴的算法。它的直观含义是“最早进入 Cache 的先被换出”。它只关心数据进来的时间早晚，不关心数据有没有被反复使用。

- **LRU (Least Recently Used，最近最少使用)**：这是考试中最常考、也是实际中最有效的算法之一。它的直观含义是“最久没有被访问的先被换出”。它基于局部性原理，认为刚刚被访问过的数据，接下来很大概率还会被访问，所以要保护那些最近常用的数据。**做题时要注意，必须维护每一组内“最近使用顺序”，而不能凭感觉**。

- **LFU (Least Frequently Used，最不经常使用)**：它的直观含义是“访问次数最少的先被换出”。它需要记录每个数据块被访问的次数，淘汰计数最低的。

- **随机替换 (Random)**：顾名思义，就是闭着眼睛随机选一行换出。虽然看起来很不讲理，但硬件实现最简单，且在容量很大时效果并不算太差。

### 4. 写策略

当 CPU 修改了 Cache 里的某块数据后，Cache 里的数据变成了最新的，但主存里的数据还是旧的。那么，**主存里的旧数据什么时候更新呢？** 解决这个同步问题的方法，就是**写策略（Write Policy）**。

最核心的两种写策略是：**写穿法（Write-Through）**和**写回法（Write-Back）**。

#### 1. 写穿法 (Write-Through) —— “步步同步”

- **工作机制**：当 CPU 要写数据时，它会**同时**把数据写进 Cache 和主存里。

- **优点**：主存里的数据永远是最新的，不需要担心数据丢失或不一致，控制逻辑非常简单。

- **缺点**：每次写操作都要访问速度很慢的主存，导致写主存的次数非常多，严重拖慢了 CPU 的整体速度（即产生大量的写流量）。

- **附加条件**：通常不需要设置“脏位”。

#### 2. 写回法 (Write-Back) —— “拖延症与打包处理”

- **工作机制**：当 CPU 要写数据时，它**只把数据写进 Cache**，主存里的依然是旧数据。只有当这个 Cache 行因为容量满了，需要被替换（换出）时，系统才会顺便把修改过的数据写回到主存中。

- **优点**：大大减少了主存的写流量。如果 CPU 对同一个变量反复修改了 100 次，写穿法要写 100 次主存，而写回法只在最后替换时写 1 次主存。

- **缺点**：主存里的数据经常是过时的（在多核处理器中会引发严重的一致性问题），而且硬件控制更复杂。

- **核心依赖：脏位 (Dirty Bit)**。系统必须知道这个 Cache 行到底有没有被修改过。如果没改过，替换时直接丢弃就行；如果被改过，**脏位就会被标记为 1**，替换时看到脏位是 1，系统就知道必须执行写回主存的操作。

## 典型例子

### 例 1：地址字段划分

某机器主存地址 32 位，Cache 数据容量 16KB，块大小 64B，采用 4 路组相联。求 tag、index、offset 位数。

先求块内偏移：

```text
块大小 = 64B = 2^6B
offset = 6 位
```

再求 Cache 行数：

```text
Cache 行数 = 16KB / 64B
          = 2^14B / 2^6B
          = 2^8 行 = 256 行
```

4 路组相联表示每组 4 行：

```text
组数 = 256 / 4 = 64 = 2^6
index = 6 位
```

最后剩下的是 tag：

```text
tag = 32 - 6 - 6 = 20 位
```

所以地址划分为：

```text
tag 20 位 | index 6 位 | offset 6 位
```

### 例 2：组号计算

Cache 有 16 行，二路组相联，主存块号为 20，问映射到哪一组。

```text
组数 = 16 / 2 = 8 组
组号 = 20 mod 8 = 4
```

所以主存块 20 只能放在第 4 组，但在第 4 组内部可以放任意一路。

### 例 3：为什么会发生冲突

直接相联 Cache 有 8 行，主存块 0 和主存块 8 的行号都是：

```text
0 mod 8 = 0
8 mod 8 = 0
```

它们都只能放第 0 行。如果程序反复访问块 0、块 8，即使 Cache 还有其他空行，也可能不断互相替换。这就是直接相联的冲突问题。

## 常见考法

- 给 Cache 参数，求 tag、index、offset 位数。
- 给主存块号，判断映射到哪一行或哪一组。
- 给一串访问序列，模拟直接相联、组相联或全相联命中情况。
- 用 LRU/FIFO 判断替换哪一块。
- 计算命中率、缺失率或平均访问时间。
- 比较写穿法和写回法，判断是否需要脏位。
- 计算 Cache 总位数，注意 tag、有效位、脏位等附加位。

平均访问时间常见表达为：

```text
平均访问时间 = 命中时间 + 缺失率 × 缺失代价
```

它说明 Cache 的意义不只是“命中时快”，还要看缺失时要付出多大代价。

## 易错点

- **offset 看块大小，不看 Cache 总容量**。
- **index 看组数，不看主存块数**。
- **二路组相联表示每组 2 行，不是 2 组**。
- **tag 不是随便取高位，而是地址去掉 index 和 offset 后剩下的高位**。
- **替换算法只在目标组满时才需要**。
- **写回法需要脏位，写穿法通常不需要脏位**。
- **Cache 对程序员透明**：程序正常写地址，不需要知道数据是否命中 Cache。

## 小结

- Cache 题先把地址拆成 `tag | index | offset`。
- 直接相联看行号，组相联看组号，全相联只比较 tag。
- `offset = log2(块大小)`，`index = log2(组数)`，剩下才是 tag。
- LRU 题要按组维护最近使用顺序。
- 写策略重点记住：写穿立即写主存，写回替换时才写主存。
